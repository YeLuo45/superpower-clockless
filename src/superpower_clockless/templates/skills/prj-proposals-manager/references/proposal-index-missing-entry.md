# proposal-index-missing-entry

## Symptom

Cron job or agent is asked to update fields on a proposal (e.g., `Technical Expectations: pending → timeout-approved`, `Current Status: approved_for_dev → in_dev`) but `proposal-index.md` contains **no entry** for that proposal ID.

However, the proposal **does exist** in `proposals.json` (the ai-superpower data store).

## Root Cause

`proposal-index.md` is a **derived index**, not the source of truth. It is generated by `sync-proposals-to-website.py`. If a proposal was created via the API but the sync script never ran (or ran against a different output path), the index entry never gets created.

Common trigger: proposals created in the ai-superpower system without running the sync script afterward.

## Detection Checklist

Before attempting any field update on `proposal-index.md`:

1. Check if the proposal exists in `proposals.json`:
   ```bash
   grep -n "P-YYYYMMDD-XXX" /home/hermes/proposals/proposals.json
   ```

2. Check if the proposal exists in `proposal-index.md`:
   ```bash
   grep -n "P-YYYYMMDD-XXX" /home/hermes/proposals/proposal-index.md
   ```

3. If present in `proposals.json` but NOT in `proposal-index.md` → this is the missing-entry problem.

4. **Also check**: if the project (e.g., `PRJ-20260412-008 ai-subscription`) has proposals in `proposals.json` but the entire project is absent from `proposal-index.md`, run a full sync rather than per-proposal reconciliation. Projects can be missing wholesale from the index if the sync script failed mid-run.

5. The API `status` vs `stage` divergence: if API returns 404 but JSON has the entry, do NOT create a new proposal — this orphans the original. Use backup/rollback instead.

## Correct Recovery Path

**Do NOT edit `proposal-index.md` manually** to create a missing entry. Instead:

1. Run the sync script to reconcile:
   ```bash
   python3 /home/hermes/proposals/scripts/sync-proposals-to-website.py [--dry-run]
   ```

2. Verify the entry now appears in `proposal-index.md`.

3. If the entry is still missing after sync, the sync script may be writing to a different path. Check `sync-proposals-to-website.py` for the output file path configuration.

4. Only after sync confirms the entry exists should you attempt field updates.

## What NOT To Do

- Do not manually add a missing entry to `proposal-index.md` — this creates a divergence between index and data store.
- Do not assume the index is always complete — it can be stale while `proposals.json` is current.
- Do not edit `proposals.json` directly to patch missing fields — use the ai-superpower API.
- **Do not create a replacement proposal with a different auto-generated ID** — this orphans data and creates ID mismatches that propagate through the system (e.g., P-20260524-005 instead of P-20260502-017). If a proposal is truly missing from the API, investigate the data store directly, do not work around it via API creation.

## Correct index path

`proposal-index.md` is located at `/home/hermes/proposals/proposal-index.md` (NOT `~/.hermes/proposals/proposal-index.md`). When a cron job references the wrong path, the file won't be found. Always use the correct absolute path when checking for missing entries.

## Target file may not exist

`proposal-index.md` may not exist at the path referenced in the cron job task. The actual location is `/home/hermes/proposals/proposal-index.md` (not `~/.hermes/proposals/`). If the target file does not exist, the index reconciliation cannot be done — log the discrepancy and report the ghost proposal state.

## Known Sync Script Bugs

If `sync-proposals-to-website.py` fails with `name 'load_mapping' is not defined`, the script is broken and cannot reconcile the index. Workaround: use the Python urllib diagnostic below to confirm API vs JSON state, then log the discrepancy and proceed without index reconciliation.

The sync script path is referenced as: `/home/hermes/proposals/scripts/sync-proposals-to-website.py` but it does NOT exist at that location in normal operation. The only copies available are in timestamped backup directories under `/home/hermes/proposals/backups/backup_YYYYMMDD_HHMMSS/scripts/sync-proposals-to-website.py`.

To locate the script at runtime, use:
```python
import subprocess
result = subprocess.run(["find", "/home/hermes/proposals/backups", "-name", "sync-proposals-to-website.py", "-type", "f"], capture_output=True, text=True, timeout=30)
script_path = result.stdout.strip().split("\n")[-1]  # Use latest backup copy
```

When the script is missing or fails, **do NOT fall back to manual index editing** — treat the proposal as a ghost in the index and log the discrepancy. The index will be reconciled on the next successful sync run.

## Known Failure Mode: Proposal in proposals.json but 404 from API

If a cron job fires on proposal P-YYYYMMDD-XXX and the API returns 404, but `grep` confirms the proposal exists in `proposals.json`:

1. The proposal was likely created in `proposals.json` directly (outside the API workflow) or the API database lost sync with `proposals.json`.
2. **Do not** try to update via API — the API has no record.
3. Log the failure and continue — do not block on missing proposals in unattended/cron scenarios.
4. Restore via backup/rollback scripts if the proposal data is critical and the discrepancy is confirmed.

## Field Mapping Note

`proposal-index.md` entries contain fields not present in `proposals.json` directly:
- `Technical Expectations` — derived from workflow state (not a stored field in proposals.json)
- `Technical Expectations Timeout Resolution` — workflow metadata
- `Technical Stack` — workflow metadata
- `PRD Confirmation` — workflow state

These are **workflow tracking fields**, not proposal data fields. They track where a proposal is in the confirmation/timeout flow. When the proposal transitions (e.g., timeout fires), the correct approach is:
1. Update via the ai-superpower API to transition status
2. Run sync script to update index
3. If index still doesn't reflect the change, the sync script may need debugging
